# Deribit Price Service

Сервис для периодического получения index price по валютным парам `btc_usd` и `eth_usd` с криптобиржи Deribit, сохранения данных в PostgreSQL и предоставления API на FastAPI для получения истории, последней цены и выборки по диапазону дат.

## Функциональность

- получение `btc_usd` и `eth_usd` c Deribit каждую минуту
- сохранение снимков цены в PostgreSQL
- REST API для:
  - получения всей истории по тикеру
  - получения последней цены по тикеру
  - получения цен по тикеру с фильтром по диапазону дат
- использование Celery Beat для периодического запуска задач
- запуск через Docker Compose
- unit-тесты для основных компонентов

Design decisions
1. Использован метод public/get_index_price

В задании требуется получать именно index price, а не цену конкретного торгового инструмента.
Поэтому был выбран endpoint Deribit public/get_index_price.

2. Каждое получение цены сохраняется как отдельная запись

Вместо обновления одной строки сервис каждый раз сохраняет новый снимок цены с текущим timestamp.

Это позволяет:

получать всю историю

получать последнюю известную цену

фильтровать данные по диапазону дат

3. Query-параметр ticker обязателен во всех API методах

Это сделано в точном соответствии с условиями тестового задания.

4. Для периодического запуска используется Celery Beat

По условию задания периодическое получение цен должно быть реализовано через Celery.
Для запуска задачи каждую минуту используется Celery Beat как стандартный планировщик для Celery.

5. Redis используется как broker/backend для Celery

Celery требует брокер сообщений. Redis выбран как простое и распространенное решение, которое удобно запускать в Docker Compose.

6. Для клиента Deribit используется aiohttp

aiohttp выбран как асинхронный HTTP-клиент, хорошо подходящий под используемый стек.

7. Использована слоистая структура проекта

Проект разделен на:

routes

services

repositories

db/models

Это упрощает сопровождение, тестирование и делает код более читаемым.

8. Для async SQLAlchemy engine используется NullPool

Так как Celery-задача использует async-код в отдельных event loop, использование NullPool позволяет избежать проблем с повторным использованием соединений между разными loop.
